Scénario 01

Premier lancement de l'application

Un utilisateur entend parler de la plateforme pour la première fois. Il télécharge l'app. Ces premiers secondes décident s'il reste ou s'il désinstalle. Chaque milliseconde compte. L'objectif : montrer la valeur avant de demander quoi que ce soit.

Écran 1 Splash Screen

Logo centré sur fond blanc. Aucun texte superflu. Durée : 1.5 secondes maximum. En arrière-plan, l'app vérifie silencieusement si une session existe déjà.

✓
Session existante détectée → redirection directe vers la Boutique. L'utilisateur ne revoit jamais l'onboarding.
→
Aucune session → passage automatique vers l'Onboarding.
✗
Jamais de splash screen de plus de 2 secondes. L'attente frustrante fait fuir.

Écran 2 Onboarding — 3 slides

3 slides uniquement. Chacune : une image simple, une phrase courte, un sous-titre de 6 mots maximum. Pas de roman. L'utilisateur comprend en 15 secondes ce que fait l'app.

1
Slide 1 : "Des produits vérifiés, livrés chez vous" — illustration produit vérifié par agent.
2
Slide 2 : "Vendez sans vous déplacer — on vient chez vous" — illustration agent terrain.
3
Slide 3 : "Sécurisé, transparent, 100% local" — bouton Commencer.
!
Bouton "Passer" visible dès la slide 1, en haut à droite. Respecter le temps de l'utilisateur.
✗
Pas de scroll vertical dans l'onboarding. Swipe horizontal uniquement, naturel et intuitif.

Écran 3 Choix d'entrée

Deux chemins clairs. L'utilisateur sait immédiatement quoi faire. Pas de menu, pas de liste d'options. Deux boutons, deux destinations.

🛒
Acheter → Boutique directement. Aucun compte requis pour parcourir.
📦
Vendre → Formulaire de soumission. Compte requis, redirection vers connexion si non connecté.
→
Lien discret en bas : "J'ai déjà un compte — Se connecter". Petit, pas intrusif.
!
Un acheteur peut voir tous les produits SANS créer de compte. On ne force jamais l'inscription avant de montrer la valeur. Règle absolue.
Scénario 02

Inscription & Connexion

Le numéro de téléphone remplace l'email — c'est la réalité camerounaise. Tout le monde a un numéro, pas forcément une adresse email mémorisable. La vérification OTP par SMS est le standard. Le processus doit être fluide, sans friction inutile, sécurisé sans être pesant.

Écran 1 Connexion

Champs minimalistes. Numéro de téléphone + mot de passe. Rien d'autre. Le clavier numérique s'ouvre automatiquement pour le numéro.

✗
3 tentatives échouées → blocage 5 min. Message précis : "Trop de tentatives. Réessayez dans 5:00".
→
Numéro inexistant → "Ce numéro n'est pas enregistré." avec lien vers inscription.
!
Mot de passe incorrect → "Mot de passe incorrect." Jamais "identifiants incorrects" — trop vague.
✓
Icône œil sur le champ mot de passe pour afficher/masquer. Standard attendu par les utilisateurs.
→
Connexion réussie → retour là où l'utilisateur était avant. Si nouveau → Boutique.

Écran 2 Inscription — Étape 1/2

Formulaire en deux étapes pour ne pas décourager avec un écran surchargé. Étape 1 : informations de base. Les règles du mot de passe sont visibles et se valident en temps réel avec des coches vertes.

✗
Numéro déjà utilisé → "Ce numéro est déjà enregistré." avec lien Se connecter. Jamais laisser l'utilisateur dans le flou.
✓
Règles mot de passe visibles en temps réel : 8 car. min ✓, Majuscule ✓, Chiffre ✓. Coches vertes qui s'allument progressivement.
!
Aucune donnée n'est enregistrée en base avant la validation OTP. Évite les comptes fantômes.
→
Bouton "Continuer" inactif tant que tous les champs ne sont pas valides. Il devient actif progressivement.

Écran 3 Vérification OTP — Étape 2/2

4 cases séparées pour le code, pas un champ texte. Le code se colle automatiquement si reçu par SMS. Compte à rebours visible. Renvoyer disponible après expiration.

✓
Code OTP copié depuis SMS → collé automatiquement dans les 4 cases. L'utilisateur n'a rien à taper.
→
Code expire après 10 minutes. Compte à rebours visible en temps réel.
✗
Maximum 3 renvois. Après : "Trop de demandes. Contactez le support." Évite les abus.
!
Code invalide → cases rougissent avec animation légère. Message : "Code incorrect. 2 essais restants."
→
Validation réussie → compte créé, connexion automatique, redirection Boutique.
Scénario 03

Boutique & Navigation Acheteur

L'écran principal de l'application. C'est la vitrine. Si cet écran est confus, surchargé ou lent — l'utilisateur part et ne revient jamais. Principe : montrer les produits le plus vite possible, avec le minimum de friction. Chaque élément affiché doit avoir une raison d'être.

Écran 1 Accueil Boutique

Header léger. Navigation par barre du bas uniquement — pas de menu hamburger. Corps centré sur les produits. Deux blocs d'entrée : catégories rapides, puis grille produits.

→
Chargement lent → skeleton loading (rectangles animés). Jamais un écran blanc ou un spinner seul.
✗
Produit réservé → carte grisée + badge "Réservé". Non cliquable. Retiré automatiquement si vendu.
!
Hors ligne → bannière discrète en haut. L'app affiche le cache. Pas de page d'erreur blanche.
✓
Scroll infini par lot de 20 produits. Retour depuis une fiche → même position dans la liste. Jamais renvoyé au début.
→
Badge rouge sur l'icône panier si articles dedans. Badge bleu sur notifications si non lues.

Écran 2 Recherche & Filtres

Suggestions automatiques pendant la frappe. Filtres instantanés sans rechargement. Résultat vide → message humain, pas technique.

✓
Suggestions apparaissent dès 2 caractères tapés. Format : "iPhone 11 (Téléphones)".
→
Filtres s'appliquent instantanément. L'utilisateur voit les résultats changer en temps réel.
!
0 résultats → "Aucun produit pour 'xxx'. Il sera peut-être disponible bientôt." Jamais "Aucun résultat".
→
5 dernières recherches mémorisées. Affichées quand l'utilisateur retape dans la barre.
Scénario 04

Fiche Produit & Processus d'Achat

L'écran de décision. L'utilisateur dit oui ou non ici. Les informations sont dans un ordre précis, pensé pour guider naturellement vers l'achat. La transparence totale sur les défauts est notre différenciateur. Aucune surprise au moment du paiement — jamais.

Écran 1 Fiche Produit

Galerie plein écran en haut. Informations dans un ordre précis et réfléchi. Le bloc "Ce que vous devez savoir" est le plus important — jamais vide, toujours honnête.

→
Non connecté + tap "Acheter" → modal de connexion par-dessus la fiche. La fiche reste visible derrière. On ne redirige pas ailleurs.
✗
Produit réservé pendant la consultation → bouton devient "Réservé — Indisponible" en temps réel. Sans rechargement.
!
Dernière unité et quelqu'un est en cours de paiement → message discret : "Un acheteur est en cours de paiement. Agissez vite."
✓
Tap sur photo → zoom plein écran avec swipe. Minimum 3 photos par produit (face, dos, défauts).
→
Bouton "Acheter maintenant" fixe en bas, toujours visible pendant le scroll.

Écrans 2 & 3 Récapitulatif & Paiement

Avant tout paiement : récapitulatif complet. Zéro frais caché. Ce que l'utilisateur voit = ce qu'il paie. Paiement Mobile Money : Orange ou MTN. Instructions claires. États de paiement gérés.

✓
Double tap impossible : le serveur génère un ID unique de transaction avant le paiement. Zéro double débit.
→
Délai d'approbation Mobile Money : 5 minutes. Après expiration → commande annulée, stock libéré automatiquement.
✗
Paiement passe mais serveur ne répond pas → "Vérification en cours..." L'utilisateur n'est jamais débité deux fois.
!
Paiement refusé → message clair + bouton "Réessayer". Jamais une page d'erreur technique.

Écran 4 Confirmation de commande

Après paiement réussi. Numéro de commande unique. Récapitulatif compact. Deux actions possibles. Notification push envoyée simultanément.

✓
Numéro de commande généré côté serveur, format lisible : #CM-2024-0047. Jamais un UUID technique.
→
Notification push envoyée immédiatement avec le même récapitulatif.
→
Deux boutons : "Suivre ma commande" + "Retour boutique". Pas de page intermédiaire.
Scénario 05

Suivi de Commande

Après le paiement, l'acheteur entre dans une phase d'attente. C'est ici que la confiance se construit ou se détruit. Une app qui garde l'utilisateur dans le noir après paiement est une app qu'on désinstalle. Chaque changement d'état doit être communiqué proactivement, avec un message humain, pas technique.

Écran 1 Liste des commandes

Vue claire de toutes les commandes. Filtres rapides pour naviguer. Badge état coloré immédiatement lisible sans même lire le texte.

→
Tri par défaut : plus récente en haut. L'utilisateur voit immédiatement sa dernière commande.
✓
Filtres : Toutes / En cours / Livrées / Annulées. Filtre instantané, sans rechargement.
!
Aucune commande → "Vous n'avez pas encore de commande." + bouton "Voir la boutique". Jamais une liste vide sans contexte.
